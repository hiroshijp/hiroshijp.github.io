**バックエンド＆GKE設計の基礎：ストレージの種類とGoogle Cloudでの対応マップ**

インフラをTerraformでコード化し、GoやSpring Bootを使ったバックエンドシステムをGKE上に展開する際、「このデータはどこに保存すべきか」と悩む場面は非常に多いと思います。

システムがスケーラブルになるほど、適材適所なデータストアの選定がアーキテクチャの要となります。今回は、ストレージの基本的な種類をおさらいしつつ、Google CloudのサービスやKubernetesの仕組みがそれぞれどれに該当するのかを整理しました。

---

### ストレージの基本的な分類

まずはデータ保存方式の基本概念です。大きく分けて以下の種類が存在します。

* **ファイルシステム（ローカル）:** OSが直接管理するツリー状のディレクトリ構造。読み書きは非常に高速ですが、特定のサーバー（ノード）に紐付くため他からはアクセスできません。
* **NFS (Network File System):** ネットワーク経由で別サーバーのディレクトリをマウントし、複数ノードから共有できる仕組み。
* **オブジェクトストレージ:** 階層を持たず、データを「オブジェクト」としてフラットな空間に保存します。REST API経由でアクセスするためWebと相性が良く、無限にスケール可能です。
* **データベース (RDBMS / NoSQL):** データを高度に構造化し、柔軟なクエリ検索やトランザクションによる整合性担保を行うための専用システム。

### Google Cloudのサービスはどう対応する？

実際のWebアプリケーション開発（例えばオンライン学習プラットフォームなど）を想定した場合、これらは以下のようにマッピングされます。

#### Cloud Storage ＝ オブジェクトストレージ
大容量データを低コストで保存するためのデフォルトの選択肢です。
* **用途の例:** ユーザーがアップロードした画像・動画ファイル、静的アセット、システムのログアーカイブなど。
* **設計のポイント:** アプリケーションからはAPI経由でアクセスします。ディスク容量の上限を気にすることなく、ステートレスなバックエンドコンテナから容易に呼び出せます。

#### Firestore ＝ データベース（NoSQL）
状態管理や検索機能を提供するドキュメント指向のデータストアです。
* **用途の例:** ユーザーのアカウント情報、学習の進捗データ、リアルタイムな状態同期など。
* **設計のポイント:** ファイルそのものではなく、Cloud Storageに保存した画像の「公開URL」や、ビジネスロジックに必要な属性情報を一貫性を持って管理する役割を担います。

### GKEの PV / PVC の正体とは？

Kubernetes環境の運用で頻繁に設定する「PV (Persistent Volume)」と「PVC (Persistent Volume Claim)」。これらはストレージの種類そのものではなく、**コンテナに対して「ファイルシステム」を割り当てるためのK8sのインターフェース**です。

裏側で動く実体（StorageClassの設定）によって、役割が変化します。

* **GCE Persistent Disk（標準設定）の場合:**
    ローカルの**「ファイルシステム」**として振る舞います。基本的に単一のPodからのみマウントして読み書きを行うため、Pod専用の一時領域やデータベースコンテナの永続化領域として使われます。
* **Google Cloud Filestoreなどを指定した場合:**
    **「NFS」**として振る舞います。複数のPodが同時に同じディレクトリを読み書きできるようになります。あるPodがソースコードを書き出し、別のPodがそれを即座にコンテナ内で実行・評価するような、複雑な連携ワークフローを組む際に活躍します。

*※注意点: Cloud StorageやFirestoreはAPI経由でネットワーク通信を行うため、PV/PVCの仕組みを使ってPodにマウントする必要はありません。*

### まとめ

モダンなアーキテクチャ設計における基本戦略は以下の通りです。

1.  Webアプリが扱うファイル（画像など）は、**オブジェクトストレージ（Cloud Storage）**へAPI経由で保存する。
2.  構造化されたデータやユーザー情報は、**データベース（FirestoreやCloud SQL）**で管理する。
3.  複数コンテナ間で直接的なファイル共有が必要な特殊要件がある場合のみ、PV/PVC経由で**NFS（Filestore）**の導入を検討する。

それぞれの特性を理解し、Terraform等のコードで適切にインフラを組み上げることで、堅牢で拡張性の高いシステムを実現していきましょう。
