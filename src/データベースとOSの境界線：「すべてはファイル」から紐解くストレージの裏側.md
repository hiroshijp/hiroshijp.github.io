# データベースとOSの境界線：「すべてはファイル」から紐解くストレージの裏側

普段、私たちが何気なくWebアプリケーションからデータを保存しているデータベース（DB）。しかし、「そのデータは物理的にどこに、どうやって書き込まれているのか？」と聞かれると、意外とフワッとしている方も多いのではないでしょうか。

今回は、Linuxサーバー上で動くデータベースを題材に、**「データがディスクに保存されるまでの旅」**をOSの視点から紐解いていきます。

## 1. データベースも、結局は「ファイル」の集合体

MySQLやPostgreSQLといった堅牢なデータベースシステムも、Linux OSから見れば1つの巨大なアプリケーションに過ぎません。テーブルに入力されたデータは、最終的にLinux上の特定のディレクトリにある**「データファイル」**として保存されます。

データベースは、単にデータを1つの巨大なファイルにベタ書きしているわけではなく、役割ごとに複数のファイルを使い分けています。

* **データファイル:** 実際のデータ（行と列）が保存されるメインのファイル。
* **インデックスファイル:** 特定のデータを爆速で探し出すための「索引」となるファイル。
* **トランザクションログ (WAL / Redo Logs):** 「〇〇を変更した」という履歴を最速で順次書き込む、障害復旧のための命綱。
* **設定ファイル:** メモリ割り当てやポート番号などを人間が読める形式（テキスト）で記述したファイル。

設定ファイル以外は、基本的にコンピュータが最速で処理できるように最適化された**バイナリファイル**です。

## 2. フロントエンドもバックエンドも、本質は「バイト配列」

少し視点を変えてみましょう。
Webブラウザ上でJavaScriptを使ってファイルを扱う際、`Blob` や `ArrayBuffer` などを通して、ファイルを単なる「バイト（0と1）の配列」として処理しますよね。

実は、OSやデータベースの最も低いレイヤーでも全く同じことが起きています。どんなファイルも、物理的なディスク上では単なる**「バイト配列の連続」**に過ぎません。そのバイトの並びを「どう解釈するか」で、それがテキストファイルになるか、データベースのバイナリデータになるかが決まっているのです。

## 3. OSはストレージをどう見ているのか？

では、OS（Linux）は物理的なストレージ（SSDやHDD）をどのように認識し、私たちが扱う「ファイル」に変換しているのでしょうか。ここには、美しく分割されたレイヤー構造が存在します。

### 物理層：ただの巨大なバイト配列

OSの最も底に近い部分から見ると、ストレージには「ファイル」や「フォルダ」という概念は一切ありません。あるのは、数億〜数千億個の小さな箱（セクタやブロック）がズラーッと並んだだけの巨大な空間です。

### デバイスファイル層：Linuxの「すべてはファイル」哲学

Linuxは、この物理ディスクをシステム上で扱うために仮の姿を与えます。それが `/dev/sda` などの**デバイスファイル**です。OSはこのファイルに対してデータを流し込むことで、間接的に物理ディスクへ書き込みを行います。

### パーティション層：空間の分割

巨大な空間をそのまま使うのは不便なので、「ここからここまではOS用」「ここから先はデータベース用」と論理的に区切ります（例: `/dev/sda1`）。

### ファイルシステム層：ここで「ファイル」が誕生する

ここがOSの魔法です。ただのバイト配列の区画に対して、`ext4` や `xfs` といった**ファイルシステム**という型枠を流し込みます。ファイルシステムはディスク内に「iノード」という管理台帳を作り、「このファイルはディスク上の何番目のブロックにあるか」を記録します。ここで初めて、人間やアプリが認識できる「ファイル」が誕生します。

### マウント層：ディレクトリツリーへの接ぎ木

最後に、ファイルシステムが作られたパーティションを、Linuxのディレクトリツリー（例: `/var/lib/mysql`）にくっつけます。これを**マウント**と呼びます。

## まとめ：美しい分業制

データベースアプリ側は「`/var/lib/mysql` という普通のディレクトリにファイルを書き込んでいる」つもりでも、OSが裏側で「そこは `/dev/sdb1` のディスクだから、そっちの物理ディスクにバイトデータを書き込むね」と交通整理をしてくれています。

普段意識しない下回りのレイヤーですが、この**「物理ディスク → デバイスファイル → ファイルシステム → マウント」**というOSの変換プロセスを知っておくと、インフラ設計や障害時のトラブルシューティングで必ず役立つはずです。
