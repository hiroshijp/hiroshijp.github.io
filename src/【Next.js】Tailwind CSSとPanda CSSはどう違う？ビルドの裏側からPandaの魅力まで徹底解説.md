# 【Next.js】Tailwind CSSとPanda CSSはどう違う？ビルドの裏側からPandaの魅力まで徹底解説

Next.js（特にApp Router環境）でのフロントエンド開発において、スタイリング手法の選択はプロジェクトの体験を大きく左右します。

現在、圧倒的なデファクトスタンダードとなっている「**Tailwind CSS**」に対し、近年大きな注目を集めているのが「**Panda CSS**」です。どちらもNext.jsのServer Componentsと非常に相性が良いモダンなツールですが、そのアプローチは全く異なります。

この記事では、この2つのツールの「ビルド周り」と「成果物」の違いを整理しつつ、Panda CSSの魅力について深掘りして解説します。

---

> 🔰 **【CSS初心者向け Hint】そもそも「CSSツールのビルド」って何？**
> 
> 昔のWeb制作では、CSSファイル（`.css`）に直接手書きでスタイルを書いていました。しかし現代の開発では、JavaScript（Reactなど）の中にスタイルを書き、それを**ツールが自動で読み取って、最終的なCSSファイルを作り出してくれる**のが主流です。この「コードを読み取って、必要なCSSファイルを作り出す作業」のことを**ビルド**と呼びます。ツールごとに「どうやって読み取るか」が違うのが、今回の大きなテーマです。

---

## 1. コードの解析アプローチ：テキストスキャン vs 構文解析

もっとも根本的な違いは、ビルド時に**ソースコードからどのように必要なスタイルを特定するか**という点にあります。

### Tailwind CSS：「正規表現」によるテキストスキャン

Tailwindは、指定されたファイル（`.tsx`など）を単なる「テキスト文字列」として読み込みます。そして、正規表現を使って `bg-blue-500` や `text-center` といったクラス名らしき文字列を高速に抽出します。

- **メリット:** 単なるテキスト検索なので**非常に高速**。どんな言語のファイルからでも抽出できます。
    
- **デメリット:** コードの「文脈」は理解していません。そのため、JavaScriptの変数を使って `className={"bg-" + color + "-500"}` のように動的にクラス名を組み立てると、ビルドツールが文字列を検知できず、スタイルが当たらないという有名な罠があります。
    

### Panda CSS：「AST（抽象構文木）」による静的解析

Panda CSSは、TypeScriptやJavaScriptのコードを「プログラムの構造（AST）」として解析します。単なる文字列としてではなく、「この関数に、このオブジェクトが渡されている」という**コードの意味を理解して**スタイルを抽出します。

- **メリット:** TypeScriptの強力な型推論をフル活用できます。変数を使った動的な記述に対しても、Tailwindより柔軟に、かつ型安全に対応できます。
    
- **デメリット:** 構文解析を行うため、Tailwindの単純なテキストスキャンと比較すると、ビルドプロセスが一段階複雑になります。
    

---

> 🔰 **【CSS初心者向け Hint】「成果物（Artifacts）」って何？**
> 
> 成果物とは、開発者が書いたコードをビルドツールが処理した後に**「最終的に出来上がったファイル」**のことです。ブラウザは私たちが書いたReactのコードをそのまま読めないので、ツールがブラウザ用のHTML/CSS/JSに変換（＝成果物を作成）してくれます。

---

## 2. 成果物の違い：何が生成されるのか？

ビルドの結果としてプロジェクト内に何が生み出され、ブラウザに何が送られるのか。ここにも明確な違いがあります。

### Tailwind CSSの成果物

- **生成物:** **1つの最適化されたCSSファイルのみ**。
    
- **特徴:** 開発者が書いた `className="text-center"` という文字列がそのままHTMLに出力されるだけです。JavaScriptのランタイム（実行時の処理）やバンドルサイズには一切影響を与えません。非常にクリーンでシンプルです。
    

### Panda CSSの成果物

- **生成物:** **CSSファイル ＋ 生成されたJS/TSファイル群（`styled-system` ディレクトリ）**。
    
- **特徴:** Pandaはビルド時に、プロジェクト内に `styled-system` というディレクトリを作ります。ここには、プロジェクト専用の型定義や `css()` 関数などが書き出されます。開発者はこれをインポートして使います。
    

TypeScript

```
// Panda CSSでのスタイリング例
import { css } from '../styled-system/css';

export default function Button() {
  // css関数を実行すると、最終的に "text_red_500" のようなクラス名の文字列が返る
  return <button className={css({ color: 'red.500' })}>クリック</button>;
}
```

---

> 🔰 **【CSS初心者向け Hint】「ゼロランタイム」とは？**
> 
> 少し前の流行りだったCSS-in-JS（Emotionなど）は、**ブラウザ上で（ユーザーが見ている裏で）JavaScriptを使ってCSSを計算し、スタイルを作って**いました。これを「ランタイム（実行時）の処理」と呼びます。しかし、これはブラウザに負担がかかり、動作が遅くなる原因にもなります。
> 
> そこで登場したのが「ゼロランタイム」。ブラウザ上では一切CSSの計算をせず、**開発時（ビルド時）にすべて計算を終わらせて静的なCSSファイルを作っておく**技術です。これにより、サイトが爆速で表示されます。

---

## 3. Panda CSS 詳説：なぜPandaを選ぶのか？

ここまで比較してくると、「Tailwindの方が生成ファイルも少なくてシンプルじゃん」と思うかもしれません。では、なぜPanda CSSが熱狂的に支持されているのでしょうか？

それは、**「Emotionのような書き味（オブジェクト記法）」と「Tailwindのようなパフォーマンス（ゼロランタイム・Atomic CSS）」を両立している**からです。

### 魅力①：圧倒的な型安全性（TypeScriptとの統合）

Panda CSS最大の武器は「型」です。`styled-system` フォルダに自動生成される型定義のおかげで、存在しないデザイントークン（色や余白のサイズ）を使おうとすると**エディタ上で即座にエラーになります**。

Tailwindでは `text-red-999` のように存在しないクラス名を書いても、ただスタイルが当たらないだけでエラーにはなりません（VSCode拡張機能で警告は出せますが）。Pandaなら、タイポによるデザインの崩れを未然に防ぐことができます。

### 魅力②：標準搭載の強力なVariants（レシピ機能）

ボタンコンポーネントを作るとき、「基本は青（primary）、でも `variant="danger"` が渡されたら赤にする」といった状態管理は頻繁に発生します。

Tailwindではこれを綺麗に書くために `tailwind-variants` や `cva` といった外部ライブラリを組み合わせるのが一般的です。しかし、Panda CSSには**「Recipes」という機能が標準で組み込まれており、型安全にVariantsを定義**できます。

TypeScript

```
import { cva } from '../styled-system/css';

const buttonRecipe = cva({
  base: { padding: '4', borderRadius: 'md', fontWeight: 'bold' },
  variants: {
    visual: {
      solid: { bg: 'blue.500', color: 'white' },
      outline: { borderWidth: '1px', borderColor: 'blue.500' }
    },
    size: {
      sm: { fontSize: '12px' },
      lg: { fontSize: '16px' }
    }
  },
  defaultVariants: {
    visual: 'solid',
    size: 'lg'
  }
});

// 使用時：TypeScriptが補完してくれる！
<button className={buttonRecipe({ visual: 'outline', size: 'sm' })}>
  送信
</button>
```

### 魅力③：Next.js (Server Components) との完全互換

Panda CSSの `css()` 関数は、実行時には「事前に生成されたクラス名の文字列」を返すだけの極めて軽量な処理です。ブラウザ上でCSSを動的に計算しない（ゼロランタイム）ため、Next.jsのApp Router（Server Components）でも制約なく、最高速のパフォーマンスで動作します。

---

## まとめ：どちらを選ぶべきか？

最終的にブラウザに送られるCSSは、どちらも「必要最小限に最適化されたクラス群」であり、パフォーマンス面で大きな差はありません。チームの好みとプロジェクトの要件で選びましょう。

- **Tailwind CSSがおすすめな人・チーム**
    
    - 環境構築を最小限にし、設定不要ですぐに書き始めたい。
        
    - ボイラープレート（自動生成ファイル）をリポジトリに入れたくない。
        
    - HTML/JSXの中にユーティリティクラス（文字列）を直書きするスタイルが好き。
        
- **Panda CSSがおすすめな人・チーム**
    
    - CSS-in-JS（オブジェクト記法）の書き味が好きだが、パフォーマンスは落としたくない。
        
    - TypeScriptの恩恵（強力な型補完・エラー検知）をスタイリングにも効かせたい。
        
    - デザインシステムを厳格に運用し、コンポーネントのVariantsをスマートに管理したい。
        

スタイリングの世界は進化が早いため、ぜひ一度両方の書き心地を試して、自分たちのチームに最適な体験を見つけてみてください！
