XMLの属性を使用しない前提だと、
抑えておきたい違いは以下
- キー(xmlの場合はタグ名)の重複
- リストの扱い

また、似てるとこは以下
- リスト内のオブジェクトの構造(xmlの場合は同名タグの列挙)が異なるときの複雑さ

以下の内容はGeminiにまとめてもらった。

---
# 🔑 キー（タグ名）の重複について

| **項目** | **JSON** | **XML（属性なし）** |
| --- | --- | --- |
| **キーの重複** | **原則として規約違反**です。1つのオブジェクト内ではキーは一意でなければなりません。 | **タグ名（要素名）の重複は許容されます。** 構造を表すために、同じ名前の要素を複数並べることができます。 |

**詳細**

- **JSON:**  
  JSON はデータ構造を表現することを主眼としており、オブジェクト内のキーはプロパティ名として一意である必要があります。

  ```json
  {
    "name": "Alice",
    "name": "Bob" // これは規約違反（多くのパーサーがエラーとするか、どちらか一方を無視する）
  }
  ```

- **XML:**  
  XML は文書構造を表現することを主眼としており、同じ種類の情報を表すために同じタグ名の要素を複数並べるのが一般的です。

  ```xml
  <people>
    <person>Alice</person>
    <person>Bob</person>
  </people>
  ```

# 📝 リスト（配列）の扱いについて

| **項目** | **JSON** | **XML（属性なし）** |
| --- | --- | --- |
| **リスト（配列）の有無** | **あります**。`[]` で表現されます。 | **リストを直接表現する構文はありません。** |

**詳細**

- **JSON:**  
  配列（リスト）があります。順序付けられたデータの集合を明確に表現できます。

  ```json
  {
    "fruits": ["apple", "banana", "cherry"]
  }
  ```

- **XML:**  
  配列に相当する機能は、通常、同じタグ名の要素を親要素の直下に複数配置することで表現します。

  ```xml
  <fruits>
    <fruit>apple</fruit>
    <fruit>banana</fruit>
    <fruit>cherry</fruit>
  </fruits>
  ```

# 🧩 リスト内の要素の構造のばらつきについて

| **項目** | **JSON（配列内）** | **XML（子要素）** |
| --- | --- | --- |
| **要素の構造のばらつき** | **許容されます**。配列は異なる型の要素（オブジェクト、文字列、数値など）を混在させることができます。 | **タグ名がばらばらでも許容されます**。親要素の子要素として、異なるタグ名の要素を混在させることができます。 |

**詳細**

- **JSON:**  
  配列内のオブジェクトの構造がばらばらでも問題ありません。

  ```json
  {
    "items": [
      { "type": "A", "value": 1 },
      { "type": "B", "detail": "info" },
      "just a string"
    ]
  }
  ```

  構文上は自由ですが、アプリケーション側での処理のしやすさから、通常は同じ構造のデータを揃えることが多いです。

- **XML:**  
  親要素の直下に異なるタグ名の子要素を混在させても問題ありません。

  ```xml
  <data>
    <header>...</header>
    <content>...</content>
    <content>...</content>
    <footer>...</footer>
  </data>
  ```
